  shared_mutex. Bit-compatible with the Python `HashRing`.
- `cache_server/third_party/blake3/` — BLAKE3 1.5.4 portable C ref
  vendored as static lib `lethe_blake3`.
- `tools/hash_compat_driver.cpp` — CLI for cross-language

  LocalHit / RemoteHit / Miss per primary ownership.
- Build infra: C language enabled, BLAKE3 lib wired into tests,
  hash_compat_driver as its own target.

### W4 server-side (this session)

- `cache_server/src/replication.cpp` — full implementation:
  - `PeerClient` (grpc::Channel + lethe::rpc::LetheCache::Stub),
    one channel per peer, multiplexed by HTTP/2.
  - `Replicator::ReplicateOut` — fire-and-forget async push to
    R-1 replica successors. N=4 worker threads behind a bounded
    queue (depth 1024). Queue overflow drops with a counter bump.
  - `Replicator::FetchFromAny` — synchronous parallel Fetch RPC
    fan-out via `std::async`; first non-empty response wins,
    abandoning the others. 500 ms per-call deadline.
  - Connection pool (`EnsurePeerClient` / `DropPeerClient`),
    populated lazily by `LetheCache::ctor` from the static
    `seed_peers`.
  - Pool state lives in a TU-local registry mapped by
    `Replicator*` so the public header stays free of gRPC type
    leaks (same trick as `Metrics::Impl`).
- `cache_server/src/main.cpp` — `--peers node@host:port,…` CLI
  flag, parsed into `cfg.seed_peers`. Self-entry filtered out.
- `cache_server/src/cache.cpp` — Replicator instantiated in
  LetheCache ctor; Insert fires `ReplicateOut` post-local-write;
  Lookup wires read-repair branch (`replicator_->FetchFromAny`
  when we are in the route but locally missed).
- `scripts/run_3node.sh` — bring up 3 local nodes on 50051/2/3
  with shared peer spec; trap-on-EXIT cleanup, per-node logs at
  `/tmp/lethe-node{0,1,2}.log`.

### W4 client-side (this session)

- `client/lethe_client/client.py` — multi-node routing:
  - Constructor accepts `peers=[(node_id, address), …]`; builds a
    local `HashRing` (bit-compatible with C++ Router).
  - Lookup partitions block_ids by primary owner, issues ONE RPC
    per distinct primary (block batching is non-optional for
    long-context throughput).
  - Transparent fetch on RemoteHit: when the server returns
    RemoteHit, client auto-dispatches Fetch against `source_node`
    and surfaces bytes in `LookupResult.fetched`.
  - Retry policy: 3 attempts, 50 ms × 2^attempt backoff,
    UNAVAILABLE / DEADLINE_EXCEEDED only.
  - Never raises into the caller: terminal RPC failure → Miss /
    None. CLAUDE.md rule 2 holds.
- `epoch_watcher.py` — already consumed PeerStatus list as of W0
  (see `docs/decisions/W3_checkpoint.md`); no W4 change needed.

### Decisions appended to `docs/DECISIONS.md`

- Async replication policy realized: fire-and-forget, bounded queue.
- gRPC connection pool: one channel per peer, lazy.
- Client retry policy: 3 × 50 ms × 2^attempt, UNAVAILABLE-only.
- Read-repair semantics on non-replica node: RemoteHit, never proxy.

## Open questions for W5-6

1. **Replication policy under realistic load.** The W4 prompt asked
   to surface if "drops happen at anything other than synthetic
   load." We did the bounded-queue design but didn't measure under
   any realistic Insert load. W5-6's RDMA work changes the per-
   peer push cost, so re-measuring then is the natural moment.
2. **Connection pool sizing.** 1 channel per peer was the W4 call;
   if W5-6 hits HTTP/2's per-connection stream cap (default 100
   concurrent streams), bump to 2-4 channels per peer and add a
   simple round-robin selector.
3. **Read-repair amplification.** When a node misses on a block it
   replicates, we do FetchFromAny → write locally. If many requests
   for the same block arrive concurrently, we may do many parallel
   FetchFromAny calls (no per-block in-flight de-duplication). W5-6
   measurement may show this as a tail-latency contributor.
4. **The 3-node smoke harness.** Whatever broke the smoke needs to
   be debugged in interactive WSL before relying on integration
   tests in this style. W5-6 will produce more multi-process tests;
   the harness mechanics need to be solid.

## One thing to do differently

**Write the test harness first.** I wrote the implementation, then
tried to spin up the smoke and got a silent hang I couldn't debug
inside the session. If I'd built the 3-node spin-up + tiny RPC
checker FIRST (as a no-op against a stub `lethe_server`), the
mechanical bugs (port races, PID cleanup, bash backgrounding
quirks) would have surfaced before the implementation made them
hard to bisect. Same lesson as W1.4's "write the byte-identical
canary before the GPU run."
