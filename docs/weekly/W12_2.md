# W12.2 — RDMA on real InfiniBand (PACE)

Upgrade the transport story from "RDMA architected" (W5-6 shipped the
`KvTransport` abstraction with `IbverbsTransport` as an abort-stub, because
WSL2's kernel had no InfiniBand subsystem) to "RDMA implemented and
functionally validated on InfiniBand." Hard-gated like W5-6 because ibverbs is
a known silent-hang risk.

All work on PACE ICE: NVIDIA L40S nodes, Mellanox **ConnectX-7** HCAs
(`mlx5_0`, `link_layer: InfiniBand`, `PORT_ACTIVE`, MTU 4096).

## Phase 0 — IB viability gate: PASS

- **Real IB hardware:** `ibv_devinfo` on two compute nodes shows `mlx5_0`
  `PORT_ACTIVE` InfiniBand. Genuine RDMA, not SoftRoCE.
- **Verbs libs + headers:** `infiniband/verbs.h`, `rdma/rdma_cma.h`,
  `libibverbs.so`, `librdmacm.so` all present.
- **Multi-node feasible:** ice-gpu had ~20 idle nodes, 0 pending; a 2-node job
  scheduled in ~10 s.
- **Fabric RDMA ceiling (independent of our code):** `ib_write_bw` between two
  nodes, 1 MiB messages, sustained **11.4 GiB/s (~96 Gbps)**. This is the
  hardware capability the verbs stack exposes; it de-risks Phase 1 and is the
  ceiling any transport built on this fabric can approach.
- **Decision: Option B** (multi-node). Dev Phase 1/2 on one node (correctness,
  cheap alloc); reserve a 2-node alloc for the Phase 3 wire perf.

## Phase 1 — implement `ibverbs_transport.cpp`: DONE

Filled in the W5-6 stub against the existing `KvTransport` interface — one
class, no changes to Replicator / routing / the interface shape.

Design (the simplest correct shape under the time box):
- **Connection: `rdma_cm`** (`rdma_resolve_addr`/`resolve_route`/`connect`).
  IPoIB is present (ib0 has an IP), so rdma_cm derives the IB path/GID/MTU —
  far less silent-hang surface than a hand-built QP state machine on IB. One
  RC QP per peer.
- **Data: two-sided SEND/RECV.** `Send` copies `[MsgHeader|payload]` into a
  registered buffer and `ibv_post_send`; the receiver's pre-posted RECV
  completes, we parse the header and hand the bytes to `on_receive`, which
  `cache.cpp` lands via `LetheCache::Insert` — the same path the gRPC Insert
  handler uses, so RDMA receives are a drop-in for gRPC pushes.
- **Fetch (read-repair pull) delegates to an internal `GrpcStreamTransport`.**
  Answering a fetch over RDMA would need the block store inside the transport;
  the throughput-critical path is the push (`Send`: replication +
  re-replication + prefill→decode), so RDMA goes there. This is a deliberate
  scope line, documented at the call site. **Consequence (see Phase 3):** the
  vLLM connector's KV *loads* are pulls, so they would not be accelerated by
  this Send-only RDMA.
- Runtime-selected by `LETHE_USE_RDMA` (one binary serves the gRPC-vs-RDMA
  A/B); peer ib0 endpoints from `LETHE_RDMA_PEERS`/`LISTEN`; per-buffer size
  from `LETHE_RDMA_MSG_BYTES`. Per-connection completion-polling thread
  (CLAUDE.md threading invariant). Lazy dial on first Send (no startup race).

Builds clean with `-DLETHE_ENABLE_RDMA=ON`; links `libibverbs`/`librdmacm`.
Default (RDMA=OFF) builds are unchanged and still ship `GrpcStreamTransport`.

Two bugs found and fixed during bring-up (both classic ibverbs):
1. **OOM.** Buffers were `cfg.send_buffer_bytes` = 16 MiB each × (32 recv + 16
   send) per conn × several conns × 3 procs ≈ 9 GiB of *pinned* memory → SLURM
   OOM-killed the step. Fixed: 1 MiB default per-buffer (env-overridable for
   the large-block microbench), fewer buffers.
2. **`rnr_retry_count` unset.** A burst of ~40 re-replication sends after a
   node kill exhausted the receiver's posted RECVs → RNR NAK → with
   rnr_retry=0 the QP errored and sends failed permanently (24/60 blocks
   stuck). Fixed: `rnr_retry_count=7` on both connect and accept.

## Phase 2 — functional validation over RDMA: PASS

Run process-based on the IB node (the docker-based W4 integration test and
W11 chaos suite can't run RDMA on PACE — there is no docker on the IB
hardware — so the equivalent invariant checks were run process-based against
the RDMA binary):

- **Happy path (INV-1, INV-6):** 60 routed inserts → every block at R=2 over
  RDMA, `under_replicated=0`, `corrupt=0` (bytes match BlockId). Transport
  stats: every posted Send completed OK, no drops.
- **Failure path (INV-1 under failure, INV-3, INV-6 through re-replication):**
  hard-kill (SIGKILL) a node → survivors detect, re-replicate the deficit over
  RDMA → `under_replicated=0`, `corrupt=0`. `done_err=0` after the RNR fix.
- **TSan:** built `-fsanitize=thread` with RDMA, ran the functest under it.
  **Zero races in the RDMA data path** (poll threads, Send, recv handling,
  buffer management — `rdma_datapath_frames=0` across all nodes). The ~568
  warnings are all in gRPC/absl library internals (the Heartbeat control-plane
  RPC + gRPC thread/channel creation, reached via the gRPC fallback) —
  uninstrumented-third-party noise present regardless of transport, not
  introduced by the RDMA code.

INV-6 (corruption) was the one to fear with RDMA zero-copy buffer management;
it held across every run, happy and failure.

## Phase 3 — wire perf: NOT LANDED (attempted; honest stop)

Two parts, both blocked, for independent reasons:

1. **2-node throughput microbench (RDMA vs gRPC Send, 1/4/16 MB, median-of-3).**
   Built and submitted (job 5345903, 2-node alloc). The bespoke 2-node
   orchestration (backgrounded `srun` servers on each node + a driver `srun`)
   did not produce an output file, and **before it could be debugged the SSH
   ControlMaster degraded into a broken state** (accepted channels, returned
   exit 0, but executed nothing / returned no output). With the connection
   unusable I could not retrieve results or iterate. The job is past its 45-min
   walltime, so no compute is being consumed. **The RDMA-vs-gRPC system
   throughput ratio is therefore not measured.** What *is* measured is the
   fabric ceiling: `ib_write_bw` = 11.4 GiB/s (Phase 0), vs gRPC-over-TCP which
   on this fabric runs over IPoIB/ethernet and is far slower — but that is the
   raw-tool number, not our transport, and is reported as such.

2. **Bigger-model TTFT (does Lethe-over-RDMA beat recompute?).** Blocked
   *independently of the connection*, for two reasons that would remain even
   with a perfect microbench:
   - The connector KV-layout limit from W12 still forces gemma-3-1b
     (gemma-3-4b/12b raise on load; Llama-3.1-8B is gated). At 1B, recompute is
     cheap, so the latency regime is unchanged.
   - This RDMA implementation accelerates the **push** path (`Send`), but the
     connector loads KV via **Fetch** (pull), which here delegates to gRPC. So
     even on a bigger model, the connector's load latency would not benefit
     from this Send-only RDMA. Closing the TTFT gap needs RDMA on the *fetch*
     path **and** a bigger model — a larger scope than W12.2.

   So the W12 finding stands unchanged: the win is **capacity**, not latency.

## Outcome

- **Bullet upgrade achieved:** "RDMA architected" → **"RDMA data path
  implemented and functionally validated on InfiniBand (ConnectX-7): R=2
  replication and failover-recovery with zero corruption/loss, TSan-clean on
  the completion-polling path."** This is the Option-A outcome, reached on
  multi-node-capable hardware.
- **Not claimed:** a measured RDMA-vs-gRPC system throughput speedup (the
  2-node run didn't land), and any latency/TTFT win (independently blocked).
  The fabric's RDMA bandwidth ceiling is 11.4 GiB/s by `ib_write_bw`.
- **To finish Phase 3 later:** with a stable connection, re-run the 2-node
  throughput job (or simplify the orchestration), and — for the latency win —
  implement RDMA on the Fetch path and obtain a larger model.
