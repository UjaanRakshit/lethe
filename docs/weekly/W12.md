# W12 — Benchmark sweep, design-doc finalization, polish

The final milestone: the evidence package. Goal was a defensible,
measured crossover (Lethe vs. a single-node vLLM with its native prefix
cache ON), a recovery curve, and a clean repo — not headline-chasing.

All numbers here were measured on PACE (Georgia Tech) ICE, one NVIDIA L40S
(48 GB) per job, `--exclusive`. Model: `google/gemma-3-1b-it`, vLLM 0.19.1,
`enforce_eager`, `block_size=16`, greedy. The C++ server was built with
gcc-12.3 against a conda gRPC 1.51.1 / protobuf 3.21.12 stack.

## Why gemma-3-1b (and not Llama-3.1-8B)

The intended sweep model was Llama-3.1-8B. `model_info` resolved but the
weight download returned `403 GatedRepoError` — access was not actually
granted, so it was unusable in the window. Gemma-3 is gated too but the
grant *was* live (verified with a real `config.json` fetch, since
`model_info` succeeds even without download rights). gemma-3-4b/12b then
failed in the connector: their KV layout is `(num_pages, 2, page_size, …)`
versus the connector's expected `(2, num_pages, …)`, raising on load. Only
the text-only **gemma-3-1b** has the compatible layout. This is the central
caveat on the whole sweep: a 1B model is small enough that **recompute is
cheap**, which changes the latency story (below). The hit-rate / capacity
story is model-independent and stands.

## Phase 1 — the crossover sweep

### Setup

Two configurations, identical workload, identical GPU and
`gpu_memory_utilization` (0.12 → the single-node KV budget):

- **Config A — baseline.** One vLLM, native prefix caching **ON**. This is
  the honest baseline (CLAUDE.md rule 1): native cache on, working set sized
  to exceed one node's KV memory.
- **Config B — Lethe.** Same vLLM + `LetheCacheConnector` against three
  loopback `lethe_server` nodes (R=2), Lethe DRAM raised to 6 GiB/node via
  `LETHE_DRAM_BYTES` so the overflow that spills out of the GPU prefix cache
  actually lands in Lethe instead of being evicted there too.

Workload: `n_distinct` random 256-token prefixes (16 blocks each), each
replayed 3× interleaved; hit rate measured on the warm passes
(`RequestOutput.num_cached_tokens / prompt tokens`). WSS is swept by
`n_distinct` ∈ {32, 64, 128, 256, 512, 1024} = roughly {0.25, 0.5, 1, 2, 4,
8}× the single-node budget. **Median of 3 independent runs** per point.

Prefixes are synthetic, not ShareGPT (see DECISIONS 2026-05-29): a cache
hit depends only on whether the working set fits, not on token *content*,
so synthetic prefixes isolate the capacity effect that ShareGPT's natural
prefix-sharing would confound. A ShareGPT replay is a deferred follow-up.

### Results (median of 3)

| WSS (distinct prefixes / tokens) | ~× budget | A native hit | B Lethe hit | A TTFT p50 | B TTFT p50 | Lethe blocks served |
| --- | --- | --- | --- | --- | --- | --- |
| 32  / 8 192   | 0.25× | 0.988 | 0.988 | 23.1 ms | 53.9 ms | 67 |
| 64  / 16 384  | 0.5×  | 0.988 | 0.988 | 23.0 ms | 52.6 ms | 120 |
| 128 / 32 768  | 1×    | 0.988 | 0.988 | 23.2 ms | 52.6 ms | 246 |
| 256 / 65 536  | 1×    | 0.988 | 0.988 | 23.2 ms | 53.2 ms | 495 |
| **512 / 131 072** | **2×** | **0.000** | **0.988** | 23.1 ms | 81.4 ms | 17 320 |
| **1024 / 262 144** | **4×** | **0.000** | **0.852** | 23.2 ms | 80.3 ms | 30 091 |

(reps were tight: e.g. B@1024 = [0.8534, 0.8517, 0.8504].)

### What this shows — and what it does not

**The capacity crossover is real and clean.** Native single-node prefix
caching holds 98.8% hit rate up to its KV budget (≈256 prefixes / 65 k
tokens here) and then **collapses to 0%** once the working set exceeds it
(512+ prefixes). Lethe **sustains 98.8% at 2× and 85.2% at 4×**, serving
tens of thousands of KV blocks from the distributed tier. This is the
distributed-systems claim the project is about: the cache holds several
times what one node can.

**The capacity win does NOT convert to a latency win at this scale, and we
do not pretend it does (CLAUDE.md rule 5).** Native TTFT is flat ~23 ms even
at 0% hit rate, because recomputing a 256-token prefill on a 1B model is
*cheaper* than a network KV fetch. Lethe's loopback fetch is 53–81 ms. So at
1B + loopback, Lethe trades latency for capacity. The latency benefit of a
KV cache only appears when per-token recompute cost exceeds fetch cost —
i.e. a larger model and/or RDMA, neither of which was reachable in the
window (model gating + connector KV-layout limit; RDMA is W5–6 SoftRoCE, not
real IB). An 8B+ model would recompute that prefill in hundreds of ms, well
above the fetch, and flip the TTFT sign. We measured 1B; we report 1B.

### Recovery curve (PACE loopback, median of 3)

Separate from the sweep: time from a node kill to full R=2 reconvergence,
three loopback `lethe_server` processes.

| working set | recovery | raw reps |
| --- | --- | --- |
| 200 blocks  | 3.7 s  | 3.2 / 3.7 / 3.8 |
| 500 blocks  | 4.4 s  | 3.9 / 4.4 / 5.4 |
| 1000 blocks | 7.6 s  | 6.2 / 7.6 / 8.2 |
| 2000 blocks | 12.0 s | 10.9 / 12.0 / 13.8 |

Reconcile with W11.1 (docker bridge, 13–22 s, ~flat): loopback strips the
bridge's fixed per-RPC latency, cutting recovery 3–4× and **revealing the
per-block re-replication cost** that the bridge overhead had masked — so on
loopback recovery grows with N (≈3 s detection floor + a drain that scales),
where on the bridge a large fixed cost dominated and looked flat. Both are
honest measurements of different environments; the ~3 s detection floor
(`dead_after`) is consistent across both. Neither contradicts the spine's
"3 s detect + re-replication" split; only the long-retired flat 3.5 s
end-to-end claim.

## Resume bullet (measured)

> **Lethe — distributed prefix-aware KV cache for disaggregated LLM serving**
> (C++20/gRPC, 3-node, R=2). Built the primitives from current production
> papers (Mooncake, DistServe, LMCache): BLAKE3 prefix-chained block IDs over a
> consistent-hash ring (128 vnodes/peer), tiered HBM/DRAM/SSD with SIEVE
> eviction, heartbeat membership with automatic re-replication, and a vLLM
> 0.19.1 KV-transfer connector. Measured on an NVIDIA L40S (gemma-3-1b,
> median-of-3): **sustains 85–99% prefix-cache hit rate at 2–4× a single node's
> KV capacity, where vLLM's native single-node prefix cache collapses to 0%**;
> full R=2 reconvergence after a node kill in 3.7–12 s across 200–2000 blocks.

One-line variant:

> Built a 3-node distributed prefix-aware KV cache (C++20/gRPC, R=2, tiered
> storage, vLLM connector) that holds 2–4× a single node's KV working set at a
> sustained 85–99% cache hit rate where vLLM's native prefix cache drops to 0%.

Do not pair these with a TTFT claim: at 1B/loopback the capacity win does not
lower TTFT (recompute < fetch). State capacity, not latency.
