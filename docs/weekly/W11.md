# W11 closeout — chaos suite + invariant verification under failure injection

**Status: W11 verified, ready for W12.** A five-scenario chaos suite
deliberately breaks the running 3-node cluster and asserts six
invariants by *behavior* (not by trusting the system's own metrics).
The suite runs in ~63s (budget: 5 min), starts from a fresh cluster, and
exits naming any failing `scenario:invariant`. The headline result:
**failover is SAFE — no data loss, no corruption, no split-brain — but
recovery to R=2 is ~2× the documented 3.5s budget, and single-pass
re-replication is bounded by `kBoundedScan=256`.**

## What ships

| File | Role |
|---|---|
| `chaos/invariants.py` | `ClusterProbe` + the six invariant checks + five scenario runners + CLI (`--scenario {sigkill,restart,pause,partition,packet_loss,packet_loss_heavy,all}`) |
| `chaos/kill_node.py` | `docker kill -s KILL/-s TERM` / `docker pause`, revive, `is_running` |
| `chaos/partition.py` | bidirectional `iptables -A OUTPUT -d <peer> -j DROP`; idempotent heal |
| `chaos/packet_loss.py` | netem if present, else iptables `-m statistic` random drop (WSL2 kernel has no `sch_netem`) |
| `chaos/run_suite.sh` | fresh `compose down`+`up`, readiness wait, 5 scenarios sequential, <5 min, exit names the failed `scenario:INV` |

## The invariants

| | Claim | Verdict |
|---|---|---|
| INV-1 | No data loss — every block on ≥1 surviving node | **PASS** (all scenarios) |
| INV-2 | Failure detected — a survivor's `cluster_epoch` bumps within `dead_after`+slack | **PASS** (~3.0–3.25s) |
| INV-3 | Failover converges to R=2 within budget | **WARN** — converges at ~7–9s, over the 3.5s target |
| INV-4 | No stale routing — no survivor returns a RemoteHit→dead node after detection | **PASS** |
| INV-5 | Load path stays alive — ring hit-rate never zeroes | **PASS** (min 0.71–1.0) |
| INV-6 | No corruption — served bytes match the BlockId | **PASS** |

## How verification works (and why not Prometheus)

The W0 stub polled Prometheus PromQL. That can't work: the scrape
interval is 5s (too coarse for a 3.5s budget), `lethe_replicas_under_target`
is the *dispatched count* of the last membership change (never resets),
and `lethe_failover_recovery_seconds` has no call site (always empty).

So the suite verifies *behaviorally*. It inserts a known corpus
**routed to each block's primary** (a true R=2 baseline — inserting all
via one node leaves ~⅓ at R=1, see DECISIONS) and then probes each
surviving node with a **local-only Fetch** (`FetchLocal`, no peer
recursion): a hit means that node physically holds the bytes, so the
per-block replica count is exact. The only metric trusted is
`lethe_cluster_epoch`, scraped sub-second off each node's /metrics. This
is deliberate — the W9 lesson was *don't trust a component to report its
own health*; ground-truth fetch-probing is the antidote.

## The five scenarios

1. **sigkill** — hard-kill node2, leave dead. Exercises all six invariants.
2. **restart** — hard-kill, restart, verify clean rejoin + reconverge.
3. **pause** — `docker pause` > `dead_after`, unpause; detected dead then rejoins.
4. **partition** — iptables split node1⇄node2 (node0 bridges). Divergent
   ring views are EXPECTED under the no-consensus design; the asserted
   claims are no loss, no corruption, and that **both isolated sides
   detect the peer's RETURN after heal** (a further epoch advance) while
   the bridging node never churns its view.
5. **packet_loss** — 5% loss on node2; must NOT be falsely declared dead
   (15 missed heartbeats over `dead_after`; P(all lost) ≈ 0 at 5%), load
   path stays alive, no corruption.

## The audit — real bug vs. expected transient

The suite came back green on the first full run. Per the brief, that
demands an audit (W8 found 5 bugs; an all-green chaos run is suspicious).
Forcing a true R=2 baseline exposed an apparent **permanent** R=1 residual
after a single death (2–14 of 48 blocks stuck, never healing, and not
healed by a rejoin either). Root-causing it:

- The `lethe_replicas_under_target` gauge read **256** — exactly
  `kBoundedScan`. The cluster had accumulated *hundreds* of blocks across
  prior runs (blocks are never deleted), so a single re-replication pass
  hit its 256-block cap and skipped the fresh corpus's blocks.
- On a **fresh cluster** (empty stores, only 48 blocks) the dispatch is
  uncapped (48/34) and re-replication **fully converges to R=2** — at
  ~7s, not 3.5s.

So the "permanent residual" was a **test artifact**, not a fundamental
bug. The two genuine findings:

**Finding A — recovery ~2× the budget.** Detection is honest (~3.0s);
re-replication then drains its fire-and-forget push queue over ~4–6s.
The budget's "500ms re-replication" slice is optimistic. INV-3 reports
this as a WARN (converges, but over target), not a failure — chasing the
exact 3.5s edge would only make the suite flaky.

**Finding B — `kBoundedScan=256` bounds single-pass completeness.** A node
holding >256 blocks won't fully re-replicate from one death. The
`TriggerReReplication` comment says the rest is "picked up on the next
eviction-loop tick" — but **`eviction.cpp` never calls
`TriggerReReplication`** (sole caller is `OnMembershipChange`). There is
no periodic re-trigger, so for a large working set under a single
permanent death, some blocks remain at R=1 until an unrelated membership
event. Safety (INV-1) still holds — those blocks survive on one node —
but a subsequent death of that node would lose them.

Both findings are **surfaced, not fixed**: changing replication/eviction
is out of W11 scope. See DECISIONS (2026-05-28, W11 finding) for the
recommended follow-up (revise the budget or batch pushes; wire a periodic
re-replication sweep / retry dropped pushes).

## Environment notes

- Runs against the W10 docker stack. The WSL2 docker kernel has no
  `sch_netem`, so `packet_loss.py` falls back to an iptables `statistic`
  random drop on INPUT+OUTPUT (netem is still tried first for kernels
  that have it).
- The suite starts fresh (`compose down`+`up`) so accumulated state never
  masks convergence; corpus (48 blocks, ~16/node) stays well under the
  256 cap within a run.

## What did NOT change

No server code. No changes to R=2, the proto, routing, replication, or
eviction. No consensus added to "fix" the partition behavior (divergent
views under a partition are the designed no-consensus outcome). The
findings are reported; the fixes are deferred with a written
recommendation.
