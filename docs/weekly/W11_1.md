# W11.1 — fix the dead re-replication re-trigger; restate the recovery budget

**Status: [[STATUS]].** Focused follow-up to W11 (not a milestone). Fixes
Finding B (re-replication completeness) and restates Finding A (the recovery
budget) honestly against a measured scaling curve. Also closes the two W10
metric holes W11 surfaced.

## Finding B — fixed: re-replication now covers the whole working set

**The bug.** `TriggerReReplication` dispatched at most `kBoundedScan=256`
blocks per pass and nothing re-triggered. Its comment claimed the remainder
was "picked up on the next eviction-loop tick," but `eviction.cpp` never
called it — the sole caller was `membership.cpp::OnMembershipChange`. So a
working set > 256 blocks per node stayed under-replicated forever after a
single permanent node death. Latent since W8 (the W8 failover test used 200
blocks, under the bound — it never exercised the overflow).

**The fix (narrow — "make the comment true").** `TriggerReReplication` now
builds the FULL candidate set (every in-route block the node holds under the
new ring, no cap) as a *re-replication round*. A dedicated low-frequency
sweep thread **owned by the Replicator** (`kReReplicationSweepInterval =
250ms`) drains the round in `kReReplicationBatch = 256`-sized, **queue-back-
pressured** chunks across successive ticks until the whole round is covered.
The first batch fires inline so recovery doesn't wait a tick.

Chosen a Replicator-owned timer over wiring the Evictor (the brief allowed
either): cohesion (no `Evictor → Replicator` edge), trivially-correct
lifetime (joined in `~Replicator` while its deps outlive it), and a smaller
race surface (sweep + worker pool share one Replicator-internal lock
discipline). See DECISIONS (2026-05-28, W11.1 Finding B fix).

**Verified.** A 600-block working set (> the 256 cap), routed to a genuine
R=2 baseline, fully reconverges to R=2 after a single permanent death:
**0/600 residual**. Pre-fix, ~340 blocks stayed at R=1 forever. The chaos
suite's new `large` scenario (`LARGE_CORPUS_SIZE=600`) is exactly this
regression test — it would fail before W11.1 and passes after.

## Finding A — restated: recovery scales with the working set

The W0 budget ("3s detection + 500ms re-replication = 3.5s") was only ever
true at tiny N. Detection is honest and size-independent (~3.0s = `dead_after`).
Re-replication time is dominated by the push-queue drain, which scales with
the working set (a synchronous unary `Insert` per push, 4 workers, and an
idempotent over-push of every in-route block).

### Measured recovery curve (docker bridge, fresh cluster, kill → full R=2)

| working set | recovery | residual | effective rate |
| ----------- | -------- | -------- | -------------- |
| 200 blocks  | 13.4s    | 0/200    | ~15 blk/s      |
| 500 blocks  | 21.8s    | 0/500    | ~23 blk/s      |
| 1000 blocks | 22.0s    | 0/1000   | ~45 blk/s      |
| 2000 blocks | 15.3s    | 0/2000   | ~131 blk/s     |

**Two things to read off this.** (1) **0 residual at every size** —
completeness holds well past the 256 cap (Finding B fixed). (2) Recovery is
**roughly flat (~13–22s), not linear** in N: 2000 blocks reconverged *faster*
than 1000 (run-to-run noise), and the effective drain rate *climbs* with N
(15 → 131 blk/s) because a large fixed cost (~13s of round latency + queue
drain beyond the ~3s detection) amortizes over more blocks. So recovery does
**not** blow up at scale — it is bounded ~constant, just slower than the
retired 3.5s claim. (Measured on the docker bridge; loopback — e.g. the W7-W8
`test_failover_recovery` at N=200 — is faster.)

### Restated budget

> **sub-3s failure detection; full R=2 reconvergence in ~15–22s, roughly
> independent of working-set size (200–2000 blocks)**, because re-replication
> throughput scales with load. The flat 3.5s claim is retired.

INV-3 in the chaos suite uses a **flat-envelope** budget
(`recovery_budget_ms(N) = ~3s detection + ~22s drain envelope + slack`, with a
tiny per-block hedge for N beyond the measured range) and FAILS only on
genuine non-convergence (residual > 0) — passing against measured reality, not
a cosmetic threshold. DESIGN.md and BENCHMARKS.md updated to match.

### Deferred re-replication optimizations (NOT done — out of W11.1 scope)

Each is more than a one-liner and touches replication semantics; they are the
levers if W12 needs recovery faster:
1. Scope the round to blocks the *dead* peer actually owned (reconstruct the
   old ring from `lost_peers`) instead of over-pushing every in-route block
   (~2× reduction).
2. Suppress the receiver's `Insert`-triggered `ReplicateOut` for
   replication-push ingests (kills a cascade).
3. Batch / parallelize the synchronous unary push path.
