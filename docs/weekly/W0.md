## Four new gaps surfaced during the fixing

These were discovered as second-order consequences of resolving the
original 14:

1. **HeartbeatReply was not 1:1 with proto** (became #1.5).
   Surfaced when the audit pass shifted from "is the design coherent
   internally?" to "does the design serialize cleanly to the wire?"
   The original spec used `std::vector<std::string> alive_peers`; the
   proto uses `repeated PeerStatus`. Resolved by option (b) widening
   (see `docs/DECISIONS.md`).

2. **`Membership::PeerInfo` had no source for `last_seen_epoch`.**
   Surfaced as a direct consequence of #1's resolution: the widened
   HeartbeatReply needed a uint64 cluster-epoch counter per peer, and
   `PeerInfo` only had a `steady_clock::time_point`. Different units;
   no clean derivation. Resolved by adding the field and implementing
   the minimum OnHeartbeat slice that populates it.

3. **`OnHeartbeat` signature discards proto fields 3-4.**
   Pre-existing, but surfaced loudly during the wire audit. Proto
   `HeartbeatRequest` carries `known_peers` (gossip) and `load`
   (admission control); the C++ signature is `(peer_id, peer_epoch)`.
   Documented as a known shape mismatch with a comment above
   `OnHeartbeat` in membership.hpp; the actual widening is W8 work.

4. **`GrpcStreamTransport` had no translation unit.**
   The class was declared in `kv_transport.hpp` but no `.cpp` file
   matched, so a default build (`LETHE_ENABLE_RDMA=OFF`) of
   `lethe_server` would have failed at link. Resolved by adding the
   stub `cache_server/src/grpc_stream_transport.cpp` and listing it
   unconditionally in `cache_server/CMakeLists.txt`.

## Decisions worth surfacing to a future reader

Three explicit decisions worth knowing about before reading the code.
Each is captured in detail in `docs/DECISIONS.md`:

1. **Heartbeat wire is the contract** — the C++ struct mirrors the
   proto, not the other way around. If the wire needs to change, the
   proto changes first and the C++ follows. See "Heartbeat wire format:
   option (b)" in DECISIONS.

2. **`KvTransport` is the abstract base name**, not `RdmaTransport`. A
   gRPC bidi-stream transport is a valid implementation; the abstract
   name should not bake in the technology. See "KvTransport naming."

3. **`PeerInfo` carries two time concepts on purpose** — wall-clock
   for the local suspect/dead detector and cluster-epoch for the wire.
   Different units, different consumers. See "PeerInfo two time
   concepts."

## Process notes

- The audit-then-fix-then-re-audit loop caught all four new gaps. None
  of them appeared in the initial 14; each was a consequence of fixing
  one of the originals. Cheap insurance: budget for one second pass
  after any non-trivial design change.
- The "verify by reading both side-by-side" discipline (proto next to
  C++ struct next to Python dataclass) is what surfaced the
  HeartbeatReply mismatch. Recommended whenever a wire-format type
  changes.
- The `static_assert`-based header smoke check (compiled with
  `cl /Zs`) was the cheapest way to confirm struct shape contracts
  hold without needing the full build toolchain. Worth keeping as a
  technique for future cross-language struct edits.
- Build verification was partial — Protobuf isn't installed on the
  development machine, so a full link couldn't be tried. Header
  syntax was verified; `Membership::OnHeartbeat` slice was actually
  compiled and run as a standalone binary against `membership.cpp`.
