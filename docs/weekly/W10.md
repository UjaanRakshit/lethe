Prometheus text exposition over a minimal raw-POSIX-socket HTTP thread.
Zero external dependency, no link libraries. See DECISIONS.md.

## The W9-lesson metric (acceptance D)

The whole point of W10, per the W9 retrospective: a flat-zero
`lethe_requests_total{result="hit"}` on a dashboard would have caught
the dead load path in minutes instead of two months. After a synthetic
workload (insert 20 blocks, look them up):

```
lethe_requests_total{node="w10node",op="lookup",result="hit"}  20
lethe_requests_total{node="w10node",op="lookup",result="miss"} 0
lethe_requests_total{node="w10node",op="insert",result="ok"}   20
```

Non-zero. The hit-rate panel is now the first thing on the dashboard
and `clamp_min`-guarded so it reads 0..1 cleanly.

## Metric families (emitted, match the dashboard)

```
lethe_requests_total{node,op,result}            counter
lethe_op_latency_seconds{node,op}               histogram (8 bounds + Inf)
lethe_tier_bytes{node,tier,state}               gauge
lethe_cluster_epoch{node}                        gauge
lethe_replicas_under_target{node}                gauge
lethe_stream_bytes_total{node}                   counter
lethe_eviction_blocks_total{node}                counter
lethe_eviction_bytes_freed_total{node}           counter
lethe_failover_recovery_seconds{node}            histogram
```

Record* wiring (audited per the W10 scope list):
- `cache.cpp::Lookup` → RecordLookup (hits/misses/latency) + the
  greppable JSON trace line. **The load-bearing one.**
- `cache.cpp::Insert` → RecordInsert.
- `replication.cpp` worker → RecordStreamBytes (bytes+latency per Send);
  TriggerReReplication → RecordUnderReplicated.
- `membership.cpp::OnMembershipChange` → RecordEpoch.
- `eviction.cpp::RunPassForTier` → RecordTierUsage (every scan, before
  the watermark early-return) + RecordEvictionPass.

Each of Evictor / Membership / Replicator gained a nullable `Metrics*`
ctor param (default nullptr → tests and METRICS=OFF builds no-op via
`if (metrics_)` guards). Metrics is constructed under
`#ifdef LETHE_ENABLE_METRICS` and declared before the subsystems so it
outlives the threads recording into it.

## Hot-path latency delta

Negligible. Record* calls are lock-free atomic fetch_add / store
(sub-microsecond). The per-Lookup JSON trace line is one `std::cerr`
write per RPC (not per block) — dwarfed by the gRPC round-trip it
accompanies. No measurable change to W1.4/W9 wall times or the
test_three_node timing assertion (still passes its <1s budget).

- **JSON-validated:** dashboard.json valid, 7 panels (added the "Cache

  `.dockerignore` so the multi-GB venv/build trees don't enter the
  build context.

  After those fixes the full stack comes up clean and the dashboard
  PromQL returns real data. Evidence (3-node cluster, 60 blocks

## Structured logging (acceptance F)

`cache.cpp::Lookup` emits one JSON object per Lookup RPC to stderr:
```json
{"evt":"lookup","node":"...","request_id":"...","requesting_node":"...","blocks":N,"hits":H,"misses":M,"latency_ms":X}
```
Stable keys, one line per RPC, greppable. (Read-repair is off at the
Lookup path since W8, so the W3-W4 read-repair-logging ask reduces to
this routing summary.)

## W1.4 reframe outcome (carried from W9, finalized in W10)

W9's presence-marker fix made the connector load path fire, which
exposed that W1.4's old A==B==C compared across the cache-hit/miss
boundary. W9 reframed it to STORE (A==B) + LOAD (C==D native-warm).
In W10 the LOAD gate turned out FP-fragile: W1.4's separate-process
model runs C as a single fused generate (Lethe-hit) and D as two-phase
warm-up+decode (native-hit) — a generate-structure mismatch that
drifts on the boundary-sensitive prompts [3,7,8], even though W9's
structurally-matched disagg==native passes all 10. Resolution
(user-approved): W1.4's STORE gate stays a hard assert (passes 10/10);
the LOAD gate is informational, with W9's test_disagg_token_identical
as the authoritative load-path gate. See DECISIONS.md.
