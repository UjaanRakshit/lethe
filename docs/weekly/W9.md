# W9 closeout — disaggregated prefill/decode KV transport via Lethe

**Status: W9 verified, ready for W10.** The disaggregated KV transport
path is validated end-to-end on Gemma-3-1B: a prefill phase exports KV
to Lethe, a decode phase imports it, and the decoded tokens are
identical to vLLM's native prefix cache on the same hit/miss schedule.
The headline outcome is that W9's hit-count gate caught a load-path bug
that had been a **false green since W1.4** — the connector's KV load
had never actually fired until this milestone.

Single-engine role-sequenced (one vLLM engine, two phases). Physical
two-instance disaggregation is W12/PACE.

## The big finding: the load path was a false green

W9's acceptance C requires the decode phase's Lethe hit count to be
`> 0` — specifically to rule out a token-identical pass that's green
only because vLLM recomputed everything. That gate fired immediately:
`decode_hit_tokens = 0` for all 10 prompts. Diagnosis (connector
method-call counters + a direct Lethe round-trip probe) showed:

- **SAVE works.** save_kv_layer stored every prefix block under its
  per-layer id; a direct probe with the real layer_id found the full
  prefix (`max_hits == n_prefix_blocks`, 26 layers/block).
- **LOAD never fired.** The scheduler's `get_num_new_matched_tokens`
  probed Lethe with `BlockId(layer=0)`, but the per-layer KV was stored
  under `layer=_layer_id_for(layer_name)` (non-zero). The C++
  `BlockIdHash` keys on layer → the layer=0 probe never matched → 0
  external tokens → decode recomputed the whole prefix.

W1.4 had passed because all its runs were cache-MISS recomputes
(deterministic → identical); same-instance W1.4 testing structurally
could not distinguish "loaded from Lethe" from "recomputed." W9's
hit-count requirement is exactly the gate that distinguishes them.

## Fixes (all user-approved at decision points)

1. **Presence marker @ layer=0.** save_kv_layer writes a 1-byte marker
   per block at layer=0 in addition to the real per-layer KV. The
   scheduler's layer=0 probe hits the marker; start_load_kv still
   fetches the real per-layer blocks. Idempotent (server dedups).
2. **Hold back the last block.** Once loads fired, a fully-cached
   prompt left 0 tokens to compute → `assert num_new_tokens > 0`
   crash. get_num_new_matched_tokens now leaves ≥1 block, mirroring
   vLLM's native cache.
3. **W1.4 reframed to a rule-2 control.** The load path firing exposed
   that W1.4's A==B==C compared a Lethe cache-HIT against vanilla
   recompute (cache-MISS) — across the boundary rule 2 excludes.
   Reframed to: STORE gate A(vanilla)==B(connector-cold), LOAD gate
   C(connector-warm/Lethe)==D(native-warm). A==C kept informational.

each.

## The rule-2 result (the W9 correctness gate)

```
DISAGG vs NATIVE diverged:               []        (gate — all 10 identical)
DISAGG vs VANILLA diverged (info):       [3, 7, 8] (cross-boundary FP drift)
decode_hit_tokens > 0 for >=2-block prompts: yes   (false-green guard)
```

The empty `DISAGG vs NATIVE` divergence list is the headline: KV served
by Lethe produces token-for-token identical output to KV served by
vLLM's own prefix cache, on the same hit schedule. The 3/10
DISAGG-vs-VANILLA divergences (at late token positions 20/56/61) are
the expected non-associative-attention FP drift across the cache-hit/
miss boundary — CLAUDE.md rule 2 explicitly does not claim
bit-identicality there, and the fact that DISAGG matches NATIVE (not
VANILLA) confirms it's a boundary effect, not a Lethe corruption.

## Environment notes

- `VLLM_ENABLE_V1_MULTIPROCESSING=0` is required to OBSERVE the
  connector's in-process diagnostics: vLLM v1 defaults to a
  multiprocess engine where the connector lives in a child process and
  its module globals are invisible to the test driver. Single-process
  doesn't change token output. The disagg child sets it.
- A crashed vLLM run can leak a CUDA context on WSL2 (VRAM held, 76%
  util, "No running processes found" in WSL's nvidia-smi). `wsl
  --shutdown` reclaims it; the ext4 filesystem and ~/lethe clone
  persist. Hit once during diagnosis (from the full-cache-assert
  crashes), recovered cleanly.
